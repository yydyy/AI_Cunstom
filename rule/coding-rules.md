# Coding Rules - 四条核心编码原则

## Rule 1: Think Before Coding (先问后写)

**核心原则**：不允许做沉默的假设，遇到不确定的地方必须先询问。

### 执行标准
- [ ] **遇到模糊需求时**：列出多种可能的理解，让用户选择
- [ ] **遇到缺失信息时**：明确提出需要补充的内容  
- [ ] **遇到技术选型时**：给出选项并说明利弊，不擅自决定

### 正例 vs 反例
```
❌ 错误（擅自假设）：
   用户："实现登录功能" → 直接写 JWT + Redis + OAuth...

✅ 正确（先问清楚）：
   用户："实现登录功能"
   → "请确认以下理解：
      1. 只需账号密码登录？
      2. 需要第三方登录吗？
      3. 登录态保持多久？"
```

---

## Rule 2: Simplicity First (极简优先)

**核心原则**：最小可行代码，200 行能 50 行写完就不要 200 行。

### 执行标准
- [ ] **不做过度设计**：不加预测性功能
- [ ] **不做不必要抽象**：能内联的就不提取
- [ ] **保持代码扁平**：优先早期返回，减少嵌套

### 正例 vs 反例
```python
❌ 过度工程：
class UserAuth:
    def __init__(self, strategy_factory, validator_chain...): ...

✅ 极简实现：
def login(user, pwd):
    return user == "admin" and pwd == "secret"
```

---

## Rule 3: Surgical Changes (精准修改)

**核心原则**：只改任务要求的部分，不碰旁边的代码。

### 执行标准
- [ ] **范围限定**：只修改与任务直接相关的代码
- [ ] **不顺手重构**：看到丑代码 ≠ 这次要改它
- [ ] **保持风格一致**：不改别人的命名风格、代码格式

### 自检查询
修改前问自己：
1. "这段代码与当前任务相关吗？"
2. "修改它会影响其他功能吗？"  
3. "这是在解决问题还是在优化代码？"（后者→另开任务）

---

## Rule 4: Goal-Driven Execution (目标驱动)

**核心原则**：把指令转成可验证的成功标准，先写测试再写实现。

### 执行标准
- [ ] **定义成功标准**：任务完成的标准是什么？
- [ ] **先写测试/验证**：写实现前，先定义如何验证正确性
- [ ] **自测通过再交付**：提交前自己跑通验证

### 成功标准示例
| 模糊任务 | 明确的成功标准 |
|---------|--------------|
| "优化查询" | "查询耗时从 500ms 降到 100ms" |
| "修复登录 bug" | "错误密码返回 401，且不崩溃" |
| "添加导出" | "点击导出，能下载 CSV 文件" |

### 工作流程
```
接收任务 → 定义成功标准 → 编写验证 → 实现 → 自测 → 交付
```

---

## 快速检查表

**编码前**：
- [ ] 需求明确？（Rule 1）
- [ ] 成功标准可验证？（Rule 4）

**编码中**：
- [ ] 只改必要代码？（Rule 3）
- [ ] 保持极简？（Rule 2）

**编码后**：
- [ ] 跑通验证？（Rule 4）
- [ ] 无无关改动？（Rule 3）
